## 5  RAY_STATUS_STEPLIM made functional

### Problem

`RAY_STATUS_STEPLIM` was being set in `rays[ray].status` when step count hit
`STEPLIM`, but `rays[ray].steps` remained *positive* (equal to `+STEPLIM`).
All application-code filters use `steps <= 0` to exclude bad rays, so stuck
rays were passing through silently.

### Fix — applied to all 5 `propagate_*` variants

After the step-count accumulation line:

```cpp
if(steps > 0) rays[ray].steps += steps;
// NEW:
if (rays[ray].status & RAY_STATUS_STEPLIM)
    rays[ray].steps = -rays[ray].steps;   // negative steps flags a stuck/failed ray
```

Now all existing downstream filters automatically exclude STEPLIM rays without
any application-code changes.

## 6  RK45_STEPLIM — faster early termination for stuck rays

### Problem

RK45 was 1 477× slower than RK4 because ~255 rays orbiting near the photon
sphere were hitting the 10 M step hard limit.

### Diagnosis

The performance test histogram showed a clean bimodal distribution:
- Well-behaved rays: maximum ~34 223 steps
- Stuck rays: exactly at 10 000 000 (STEPLIM)
- Gap factor: ~290×

Additionally, the outer-loop step cap included:

```cpp
T step_max = abs(theta / ptheta1) / theta_precision;
```

With the source at `theta = 1e-3`, this cap was tiny and prevented the
adaptive controller from growing the step size.  The cap was **removed** from
both RK45 outer loop blocks (retained only for the horizon cap).

### Fix

**`src/raytracer/raytracer.h`:**

```cpp
// step limit for the RK45/DOPRI5 adaptive integrator
#define RK45_STEPLIM 100000
```

100 000 gives ~3× safety margin over the observed maximum for well-behaved
rays and terminates stuck rays ~100× faster than STEPLIM.

Wired into both `run_raytrace_rk45` variants (replacing STEPLIM in the
continue guard and the `propagate_rk45` call).

After fix: wall time dropped from ~155 s → ~0.9 s.

## 7  Photon-sphere separatrix diagnosis

### Observation

The emissivity plot showed RK45 deviations at two specific radial ranges:
- Bins 4–5 (r ≈ 2.75–3.36 Rg): RK45 has ~3 160 *more* rays → emissivity +19–31 %
- Bins 9–10 (r ≈ 7.49–9.15 Rg): RK45 has ~3 181 *fewer* rays → emissivity −22–37 %

### Diagnosis

The difference (3 160 excess vs 3 181 deficit = 21-ray balance, 0.7 %) shows
the *same* rays are redistributed between bins, not created or destroyed.  A
Pearson correlation of **r = 0.998** between N_ratio (RK45/RK4 ray count) and
relative emissivity deviation confirms the effect is entirely driven by ray
redistribution, not per-ray algorithmic noise.

**Initial diagnosis (later revised):** attributed to the retrograde photon
sphere separatrix — different effective step sizes in RK4 vs RK45 causing
borderline rays to orbit an extra time and land in different bins.

A diagnostic figure `emissivity_rk45_diagnosis.png` was produced showing:
1. Emissivity profiles with photon-sphere marker
2. N_ratio (RK45/RK4) vs radius
3. Scatter plot N_ratio vs rel_emis with linear fit (r = 0.998)

**Revised root cause (identified in later session):** the `COUNT_MIN = 100`
step-count guard on sign flips was the dominant driver.  After an r turning
point, the old code required 100 more steps with `rdotsq > 0` before allowing
another sign flip.  During those 100 steps the ray propagated in the wrong
radial direction.  Near the photon sphere, where trajectory fate is highly
sensitive to small changes in r, this systematic 100-step overshoot was enough
to push separatrix-grazing rays across the dividing line.  Because RK4 and
RK45 have different step sizes, the coordinate distance covered during the
100-step overshoot differed, causing the observed systematic bin migrations.

Replacing the count-based guard with the `was_positive` boolean (§11) reduced
the photon-sphere bin deviations from ±19–37 % down to ±2–5 %, confirming
the guard was the primary cause.

## 8  Tunable rk45_tol precision parameter

The DOPRI5 per-step mixed absolute/relative error tolerance was promoted from
a `static constexpr` local to a runtime-tunable member variable.

### Changes — `src/raytracer/raytracer.h`

```cpp
// private member:
T rk45_tol;   // per-step mixed abs/rel error tolerance for the DOPRI5 adaptive controller

// public methods:
void set_rk45_tol(T tol) { rk45_tol = tol; }
T    get_rk45_tol() const { return rk45_tol; }
```

### Changes — `src/raytracer/raytracer.cpp`

Constructor initializer: `, rk45_tol(T(1e-8))`

Both `propagate_rk45` variants: replaced

```cpp
static constexpr T tol = T(1e-8);
```
with
```cpp
const T tol = rk45_tol;   // tunable via set_rk45_tol()
```

### `emissivity_rk45_plot.cpp` update

Accepts an optional second argument `argv[2]` for the tolerance:

```
bin/emissivity_rk45_plot [output.csv] [tol]
```

Applied via `rk45.set_rk45_tol(rk45_tol_arg)` before `reduce()`.

## 9  Tolerance sweep — emissivity_rk45_tol_sweep.py

**File:** `src/tests/emissivity_rk45_tol_sweep.py`

Sweeps `rk45_tol` over 9 values (1e-6 → 1e-10) and plots:
1. RMS emissivity deviation (%) vs tolerance
2. Max |emissivity deviation| (%) vs tolerance
3. Total wall-clock time (s) vs tolerance

### Results

| tol | RMS dev | Max dev | Wall (s) |
|-----|---------|---------|----------|
| 1e-6 | 13.4 % | 37.0 % | 18.6 |
| 3e-7 | 12.4 % | 37.0 % | 18.6 |
| 1e-7 | 12.2 % | 36.6 % | 18.6 |
| 3e-8 | 11.9 % | 36.6 % | 18.8 |
| **1e-8** | **11.9 %** | **36.7 %** | 18.8 |
| 3e-9 | 11.8 % | 36.5 % | 18.8 |
| 1e-9 | 12.7 % | 42.2 % | 18.8 |
| 3e-10 | 12.4 % | 37.8 % | 18.5 |
| 1e-10 | 12.8 % | 42.4 % | 18.5 |

### Conclusion

The deviations are **flat across all tolerance values**.  Tightening `rk45_tol`
does not reduce the photon-sphere separatrix discrepancy.  The effect is
intrinsic to the separatrix topology: any change in step size shifts which
borderline rays orbit the photon sphere an extra time, regardless of how
accurate each individual step is.  Wall-clock time is also essentially
unchanged (the fixed-grid RK4 run dominates; RK45 is fast after `RK45_STEPLIM`).

`rk45_tol` remains useful for users needing tighter *trajectory* accuracy for
individual ray paths, but it is not a lever for closing the emissivity
difference between integrators.
