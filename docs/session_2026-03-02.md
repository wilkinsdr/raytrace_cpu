## 1  Alternative integrators discussion

Reviewed the existing two integrators:

- **`propagate()` (Euler / semi-analytic)** — first-order.  Momenta are re-derived algebraically from the conserved constants of motion (k, h, Q) at each position; only the position update is Euler.  O(step) global error.
- **`propagate_rk4()` (fixed-step RK4)** — fourth-order.  Evaluates `momentum_from_consts` at four trial positions (k1–k4), combines with standard 1/6 : 2/6 : 2/6 : 1/6 weights.  O(step⁴) global error.

Key structural observation: because k, h, Q are exact conserved quantities, integration error lives entirely in the *position* update, not in the momenta.

Known limitations identified:

| Issue | Detail |
|---|---|
| Step sizing vs integrator order | Both methods use the same `\|r−r_h\| / pr / precision` heuristic, so RK4 pays 4× the function evaluations but doesn't take credit for higher accuracy in choosing `step` |
| `COUNT_MIN = 100` sign-flip guard | Fragile near turning points; depends on step size |
| RK4 sub-step sign inheritance | k2–k4 always use k1 `rdot_sign`/`thetadot_sign`; wrong if a turning point is crossed within a step |

Alternatives considered:

- **Dormand-Prince RK45 (DOPRI5)** — adaptive embedded 4th/5th-order pair; best speed/accuracy trade-off; replaces ad-hoc `precision` heuristic with a principled tolerance; natural shrinkage near turning points and horizon.
- **DOP853** — 8th-order; excellent for smooth trajectories; overkill for most runs.
- **Symplectic integrators** — preserve null geodesic constraint `g_μν pᵘ pᵛ = 0` over many steps; relevant for photon-sphere orbits.
- **Exact turning-point detection** — analytically find zeros of R(r) and Θ(θ); eliminates `COUNT_MIN` heuristic entirely.

Decision: implement Dormand-Prince RK45 (DOPRI5) first.

## 2  Naming: RKF45 vs RK45 vs DOPRI5

`RKF45` specifically names the **Runge-Kutta-Fehlberg** method (Fehlberg 1969).  The preferred modern variant is **Dormand-Prince** (DOPRI5, 1980), which:

- Propagates with the *5th*-order solution (local extrapolation, better accuracy)
- Uses the 4th-order solution only for error estimation
- Has the **FSAL** property: k7 at step n = k1 at step n+1 (saves one derivative call per accepted step)
- Is what GSL, SciPy `odeint`, and MATLAB `ode45` all use

The new function is named `propagate_rk45` / `run_raytrace_rk45` for user simplicity, with a comment documenting the DOPRI5 algorithm.

## 3  `rk45_tol` parameter design

Options considered:

1. Separate `rk45_tol` member — cleanest semantics
2. Derive from `precision` as `tol = 1/(precision²)` — keeps one accuracy knob
3. Hard-code tolerance (1e-8) and use `precision` only for the first-step seed — simplest interface

Decision: **option 3** initially (hard-coded 1e-8, `precision` seeds first step only).  Subsequently the user added `rk45_tol` as a tunable member with `set_rk45_tol()` / `get_rk45_tol()` for the separatrix-sensitivity use-case documented in the 2026-03-01 session.

## 4  DOPRI5 implementation

New functions added to `src/raytracer/raytracer.cpp` and declared in `src/raytracer/raytracer.h`:

- `run_raytrace_rk45(r_max, theta_max, ...)` — OMP-parallel wrapper (serial for file output)
- `propagate_rk45(ray, rlim, thetalim, steplim, ...)` — theta_max stopping condition
- `run_raytrace_rk45(r_max, RayDestination*, ...)` — RayDestination wrapper
- `propagate_rk45(ray, rlim, RayDestination*, steplim, ...)` — RayDestination stopping condition

### DOPRI5 Butcher tableau (Dormand & Prince 1980, Table 2)

Nodes c = {0, 1/5, 3/10, 4/5, 8/9, 1, 1}.  7 stages; b2 = 0 so k2 is absent from the 5th-order solution.

5th-order weights: b = {35/384, 0, 500/1113, 125/192, −2187/6784, 11/84, 0}

Error coefficients (b − b*, using k7 FSAL stage):
e = {71/57600, 0, −71/16695, 71/1920, −17253/339200, 22/525, −1/40}

All coefficients verified by computing b_i − b*_i from the published 4th-order weights.  Sum of 5th-order weights checked = 1.

### Step-size control

```
h_new = h * min(5, max(0.1, 0.9 * (1/err)^(1/5)))
```

Error norm (mixed absolute/relative over r and θ):

```
sc_i = tol * (1 + max(|y_n_i|, |y_{n+1}_i|))
err  = sqrt(0.5 * ((err_r/sc_r)² + (err_θ/sc_θ)²))
```

### Design decisions

| Aspect | Decision |
|---|---|
| Angular momentum variable | `h` (consistent with existing code); step size uses `step` / `h_try` |
| FSAL | Not implemented; k1 recomputed each step (noted in comment as future optimisation) |
| Boundary clamping | `h_try` clamped to `rlim`/`thetalim`; running `step` not updated on a clamped accept |
| MIN_STEP floor | Force-accept if adaptive controller can't shrink below `MIN_STEP = 1e-3` |
| Sign-flip logic | Identical to `propagate_rk4` |

Build verified clean with `cmake --build . --target emissivity` (only pre-existing warnings).

## 5  Interface refactoring (user-applied after session)

The user refactored the public API to use an `Integrator` enum:

```cpp
enum class Integrator { Euler, RK4, RK45 };

void run_raytrace(Integrator method, T theta_max, T r_max, ...);
void run_raytrace(RayDestination<T>* dest, Integrator method, T r_max, ...);
```

`emissivity.cpp` updated to call `run_raytrace(Integrator::RK45, M_PI_2, r_max, ...)`.

A separate `RK45_STEPLIM = 100000` macro was added (vs `STEPLIM = 10000000`) to terminate stuck photon-sphere orbiting rays ~100× faster.

`rk45_tol` added as a tunable member with getter/setter.
